# Деплой обновления: статистика сущностей (символы, локации, теги)

При выкатке функционала «Сущности толкований» (таблицы `dream_interpretation_entities`, `dream_entity_daily`, админка, динамика по дням) на продакшене нужно выполнить шаги ниже.

---

## 1. Миграции

```bash
php8.3 artisan migrate --force
```

Создаются таблицы (если ещё не применены):

- `dream_interpretation_entities`
- `dream_entity_daily`

---

## 2. Первичный расчёт по всем толкованиям

Сначала нужно проиндексировать сущности из всех существующих толкований (выгрузить символы, локации, теги из JSON в таблицу):

```bash
php8.3 artisan interpretations:index-entities --chunk=500
```

Опционально перед этим проверить объём без записи:

```bash
php8.3 artisan interpretations:index-entities --dry-run
```

Затем заполнить дневную агрегацию по всем датам, которые уже есть в `dream_interpretation_entities`:

```bash
php8.3 artisan interpretations:aggregate-entity-daily --all
```

После этого в админке будут доступны: общий топ сущностей, топ за любую дату, сравнение двух дней, динамика по сущности.

---

## 3. Ежедневный сбор за «вчера» (cron)

Раз в день нужно:

1. Добавить в индекс сущности из **новых** толкований.
2. Агрегировать сущности за **вчера** в `dream_entity_daily`.

В проекте это уже запланировано в `bootstrap/app.php`:

- `interpretations:index-entities --only-new` — ежедневно.
- `interpretations:aggregate-entity-daily --yesterday` — ежедневно в **01:00**.

Чтобы расписание выполнялось, на сервере должен быть настроен cron (один раз):

```bash
* * * * * cd /path-to-project && php8.3 artisan schedule:run >> /dev/null 2>&1
```

Подставьте свой путь к проекту и при необходимости `php8.3` (или `php`).

---

## 4. Часовой пояс для «вчера»

Команда `interpretations:aggregate-entity-daily --yesterday` считает «вчера» по **часовому поясу из настроек сайта** (админка → Настройки → поле «Часовой пояс»). Если пояс там указан (например `Europe/Moscow`), агрегация за вчера будет ровно за тот календарный день в этом поясе.

Рекомендуется:

- В админке в настройках задать нужный часовой пояс (если ещё не задан).
- Убедиться, что cron вызывает `schedule:run` в подходящее время (например после полуночи по этому поясу), чтобы «вчера» уже не менялось. Запуск в 01:00 по серверу обычно подходит, если сервер в том же поясе или `schedule:run` выполняется после полуночи в нужном поясе.

Если часовой пояс в настройках не задан или невалидный, используется `config('app.timezone')` (часто `UTC`).

---

## 5. Краткий чеклист деплоя

- [ ] `git pull`, `composer install`, `npm run build` (если нужно).
- [ ] `php8.3 artisan migrate --force`
- [ ] `php8.3 artisan interpretations:index-entities --chunk=500`
- [ ] `php8.3 artisan interpretations:aggregate-entity-daily --all`
- [ ] Проверить, что cron для `schedule:run` настроен.
- [ ] В админке проверить настройку часового пояса (для корректного «вчера»).
- [ ] Очистить кэш при необходимости: `php8.3 artisan config:cache` и т.д.

После этого обновление статистики сущностей при деплое выполнено, а раз в день будут подтягиваться новые толкования и пересчитываться статистика за вчера с учётом часового пояса.
